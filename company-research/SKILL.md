# 公司尽调研究

对投资目标公司进行系统性尽调研究，构建公司画像，深度分析业务与技术，回答客户核心问题，生成投资研究报告。

**核心原则**：PE 投资研究的目标是回答"投不投"和"投多少"。技术驱动型公司的尽调必须包含技术深度分析——技术是核心资产，也是核心风险源。

---

## 工作流

### 第零步：读取配置与协作模式

先读取统一配置，再决定搜索工具和输出方式。配置策略参考：

- 搜索工具配置：`references/search-policy.md`
- 协作模式配置：`references/collaboration-policy.md`
- 飞书操作指南：`references/lark-collaboration.md`

配置读取顺序：

1. 优先读取 `config/investment-research-config.json`
2. 其次使用本 skill 运行期配置；如果缺失则首次运行时询问用户
3. 如果搜索工具或协作模式仍为空或为 `ask`，首次运行时询问用户选择，并写入 `config/investment-research-config.json`

协作模式：

| 模式 | 输入 | 输出 |
|------|------|------|
| 本地模式 | 本地文件目录 | 本地 Markdown 文件 |
| 飞书协作模式 | 本地文件 + 可选飞书云文档 | 飞书云文档 + 本地备份 |

如果配置为飞书协作模式：
1. 询问是否有飞书云文档需要读取（URL 或关键词搜索）
2. 使用 `lark-cli docs +fetch` 读取飞书文档内容
3. 与本地文件内容合并

如果飞书工具不可用或权限不足，回退到本地 Markdown 输出，不中断研究。

### 第一步：加载文档

使用文档加载器读取资料目录中的所有文档：

```bash
python scripts/doc_loader.py "资料目录路径"
```

加载后的文档缓存到 `_doc_cache/` 目录，后续步骤直接读取缓存。

**飞书文档**：如果第零步选择了飞书协作模式，将飞书文档内容与本地文件合并，统一处理。

### 第二步：内容精炼

对加载的文档进行结构化精炼，提取关键信息：

1. **公司基本信息**：公司名、成立时间、行业、融资轮次、估值
2. **核心业务**：主营业务、产品线、目标客户、商业模式
3. **财务数据**：营收、利润、增长率、关键财务指标
4. **团队信息**：创始人、核心团队、组织架构
5. **技术信息**：核心技术、技术栈、技术壁垒、研发投入

精炼结果保存到缓存，供后续步骤使用。

### 第三步：业务与技术深度分析

对公司的业务和技术进行深度分析，构建完整的公司画像。

#### 3.1 公司概况

- 公司定位与核心价值主张
- 发展历程与关键里程碑
- 股权结构与融资历史
- 组织架构与团队规模

#### 3.2 业务分析

- 主营业务与产品线
- 目标客户与市场定位
- 商业模式与收入结构
- 增长驱动与增长逻辑
- 竞争格局与竞争地位

#### 3.3 财务分析

- 营收规模与增长趋势
- 盈利能力与盈利趋势
- 现金流状况
- 关键财务比率
- 单位经济模型（如适用）

#### 3.4 核心技术深度分析

**这是技术驱动型公司尽调的核心环节。**

1. **技术架构拆解**
   - 核心技术栈：语言、框架、基础设施
   - 架构模式：单体/微服务/事件驱动/混合
   - 关键组件：自研 vs 开源，核心 vs 边缘
   - 数据流：数据采集、处理、存储、输出的完整链路
   - 技术选型理由：为什么选择这些技术？trade-off 是什么？

2. **技术护城河评估**
   - 核心技术壁垒：专利布局、算法壁垒、数据壁垒、工程经验壁垒
   - 壁垒深度：竞品绕过成本、替代方案成熟度
   - 壁垒可持续性：技术迭代速度、护城河是加宽还是收窄
   - 开源依赖风险：核心组件是否依赖开源项目？自研比例？

3. **研发能力评估**
   - 研发团队规模与背景：核心技术人员履历
   - 研发投入占比：与行业标杆对比
   - 技术迭代速度：版本发布频率、技术演进路线
   - CTO/技术负责人：行业影响力、技术视野

4. **技术差异化**
   - 与竞品的技术架构差异
   - 技术优势的具体体现（性能、成本、体验）
   - 技术劣势和改进方向
   - 技术选型的独特性和合理性

5. **技术债务与风险**
   - 技术成熟度：从 POC 到大规模生产的距离
   - 扩展性瓶颈：当前架构的扩展上限
   - 技术路线依赖：是否锁定在某个技术方向
   - 安全与合规：数据安全、隐私合规、技术合规

**技术信息搜索策略**：如果资料中技术信息不足，按 `references/search-policy.md` 选择用户配置的通用搜索工具和技术搜索工具，补充搜索公司技术博客、技术分享、开源项目、技术文档、架构设计文档、技术评测和 benchmark 对比。

### 第四步：回答客户问题

根据客户提供的核心问题或从分析中提炼的核心洞察，进行系统性回答。

#### 4.1 如果客户提供了核心问题列表

逐个回答客户的核心问题，使用分层回答机制：

| 状态 | 标记 | 含义 |
|------|------|------|
| ✅ 已解决 | `✅ 已解决` | 资料中有充分信息可以回答 |
| ⚠️ 待确认 | `⚠️ 待确认` | 有部分信息但需要进一步验证 |
| ❌ 未涉及 | `❌ 未涉及` | 资料中完全没有相关信息 |

**回答格式**：
```
### Q{n}. {问题}

**状态**：✅/⚠️/❌
**回答**：基于资料的具体回答
**证据**：支撑回答的关键信息及来源
**不确定性**：回答中不确定的部分及原因
**建议**：如需进一步确认，建议的验证方式
```

**技术问题优先回答**：如果客户问题中包含技术问题，优先处理，因为技术问题往往需要更深入的分析。

#### 4.2 如果客户未提供问题

从步骤1-3的分析中提炼5-8个核心洞察：

1. 最关键的投资亮点（1-2个）
2. 最大的投资风险（1-2个）
3. 最需要进一步验证的领域（1-2个）
4. 技术维度的核心判断（1-2个）
5. 估值相关的关键发现（1个）

### 第五步：生成报告

基于以上分析，生成结构化的投资研究报告。

报告模板参考：`references/output-template.md`

**报告结构**：

1. **执行摘要**：一页纸总结，包含投资建议
2. **公司画像**：公司概况、业务、团队
3. **财务分析**：关键财务数据和趋势
4. **技术深度分析**：技术架构、护城河、研发能力、差异化、风险
5. **客户问题回答**：逐个回答客户的核心问题
6. **核心洞察**：关键发现和判断
7. **投资建议**：建议推进/谨慎推进/暂缓推进/不建议推进

**输出方式**：
- 本地模式：写入 Markdown 文件到资料目录同级
- 飞书协作模式：
  - 追加到已有飞书文档：`lark-cli docs +update --api-version v2 --doc <token> --command append --content '<XML>'`
  - 创建新飞书文档：`lark-cli docs +create --api-version v2 --content '<XML>'`
  - 发送文档链接：`lark-cli im +messages-send --user-id <open_id> --as user --markdown "报告链接: <url>"`
  - 同时本地保存 Markdown 备份

---

## 输出契约

最终输出包含以下部分：

1. **公司画像**：结构化的公司信息摘要
2. **业务与技术深度分析**：业务分析 + 核心技术深度分析
3. **客户问题回答结果**：分层回答（✅/⚠️/❌）
4. **核心洞察**：5-8个关键发现
5. **投资建议**：带理由的投资建议

## 前置与后续

**前置技能**：
- `industry-research`（推荐）：先了解行业背景，再研究公司更有针对性

**后续技能**：
- `critical-questions`（推荐）：识别盲区、交叉验证、提出关键问题
- `competitive-analysis`（推荐）：与竞对进行深度对比分析

---

## 注意事项

- 公司研究的核心是"回答投资决策问题"，不是"罗列所有信息"
- 技术深度分析是科技公司的核心环节，不能省略或简化
- 如果资料中技术信息严重不足，应在报告中明确标注，并建议通过 critical-questions 技能进一步识别技术盲区
- 回答客户问题时，区分"事实"和"推断"，推断需标注依据和不确定性
- 投资建议必须基于分析，不能仅凭直觉
