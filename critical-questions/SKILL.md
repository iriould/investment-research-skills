# 关键问题识别与提出

对投资目标公司的资料进行系统性审查，识别投资盲区、交叉验证信息一致性，提出投决必知的关键问题。

**核心原则**：PE 投资决策的质量取决于问题的质量，而非答案的数量。技术驱动型公司尤其需要技术维度的深度验证。

---

## 工作流

### 第零步：读取配置与协作模式

先读取统一配置，再决定输出方式。配置策略参考：

- 协作模式配置：`references/collaboration-policy.md`
- 搜索工具配置：`references/search-policy.md`
- 飞书操作指南：`references/lark-collaboration.md`

配置读取顺序：

1. 优先读取 `config/investment-research-config.json`
2. 其次使用本 skill 运行期配置；如果缺失则首次运行时询问用户
3. 如果协作模式仍为空或为 `ask`，首次运行时询问用户选择，并写入 `config/investment-research-config.json`

协作模式：

| 模式 | 输入 | 输出 |
|------|------|------|
| 本地模式 | 本地文件目录 | 本地 Markdown 文件 |
| 飞书协作模式 | 本地文件 + 可选飞书云文档 | 飞书云文档 + 本地备份 |

如果配置为飞书协作模式：
1. 询问是否有飞书云文档需要读取（URL 或关键词搜索）
2. 使用 `lark-cli docs +fetch` 读取飞书文档内容
3. 与本地文件内容合并

如果飞书工具不可用或权限不足，回退到本地 Markdown 输出，不中断研究。

### 第一步：加载文档

使用文档加载器读取资料目录中的所有文档：

```bash
python scripts/doc_loader.py <资料目录路径>
```

加载后的文档缓存到 `_doc_cache/` 目录，后续步骤直接读取缓存。

**飞书文档**：如果第零步选择了飞书协作模式，将飞书文档内容与本地文件合并，统一处理。

### 第二步：投资盲区识别

运行盲区识别脚本，自动检测投资盲区：

```bash
python scripts/blindspots.py <资料目录路径>
```

脚本输出 JSON 格式的盲区列表，包含以下类型：

| 盲区类型 | 检测逻辑 | 严重性 |
|----------|---------|--------|
| 产品细节缺失 | 产品被提及但缺少功能/参数/客户反馈 | 高 |
| 行业关键指标缺失 | 行业核心指标（TAM/市占率/增速等）未涉及 | 高 |
| 客户集中度风险 | 客户数量少或完全缺失 | 高 |
| 客户信息完全缺失 | 资料中未提及任何客户信息 | 高 |
| 竞争格局模糊 | 声称有优势但未指明竞争对手 | 中 |
| 竞争格局不完整 | 仅提及1-2个竞争对手 | 中 |
| 退出路径未讨论 | 无 IPO/并购/回购信息 | 高 |
| 创始人信息不足 | 提及创始人但无背景，或完全未提及 | 中 |
| 技术架构缺失 | 声称技术优势但无架构/技术栈描述 | 高 |
| 技术性能指标缺失 | 声称技术领先但无量化指标/benchmark | 高 |
| 技术壁垒证据缺失 | 声称有壁垒但无专利/数据/具体证据 | 高 |
| 研发投入信息缺失 | 无研发团队/投入/技术负责人信息 | 中 |
| 技术路线风险未讨论 | 有技术内容但未讨论替代/颠覆风险 | 中 |

**行业识别**：脚本自动从文本中推测行业，匹配行业关键指标。如果行业未识别，使用通用指标。

**盲区优先级排序**：
1. 高严重性盲区优先处理
2. 同等严重性下，技术盲区优先（科技领域投资中技术盲区往往是最大风险源）
3. 多个盲区指向同一领域时，合并为更深层的问题

### 第三步：交叉验证

对资料中的关键信息进行一致性验证，识别矛盾和红旗信号。

#### 3.1 数字一致性验证

检查资料中的数字是否自洽：

- 营收数字：各处提及的营收是否一致（总额 vs 分项之和）
- 增长率：复合增长率与年度增长率是否匹配
- 市场规模：TAM/SAM/SOM 的计算逻辑是否合理
- 估值：估值与营收/利润的倍数关系是否在合理区间
- 技术指标：声称的性能指标与行业公开基准是否一致

#### 3.2 逻辑一致性验证

检查论述逻辑是否自洽：

- 增长逻辑：声称的增长驱动是否与行业趋势一致
- 竞争逻辑：声称的竞争优势是否与市场地位匹配
- 技术逻辑：声称的技术领先是否与研发投入匹配
- 商业逻辑：商业模式与目标客户是否匹配

#### 3.3 证据强度验证

评估关键声明的证据支撑：

| 证据等级 | 标准 | 示例 |
|----------|------|------|
| 强证据 | 第三方验证+量化数据 | 审计财报、专利证书、benchmark 测试报告 |
| 中等证据 | 公司自述+部分佐证 | 融资材料、媒体报道、客户证言 |
| 弱证据 | 仅公司自述，无外部验证 | BP 中的声明、官网描述 |
| 无证据 | 仅有结论，无任何支撑 | "技术领先"但无任何数据 |

**重点关注技术声明的证据强度**：
- "技术领先" → 是否有 benchmark/第三方测试？
- "自研核心" → 是否有专利/开源对比？
- "性能提升 X%" → 对比基线是什么？测试条件？
- "数据壁垒" → 数据规模、独有性、合规性如何？

#### 3.4 红旗检查

识别需要特别警惕的信号：

- 关键数字前后不一致
- 核心声明缺乏证据支撑
- 回避性表述（"不便透露"、"商业机密"出现在关键问题上）
- 过于完美的增长曲线
- 技术声明与行业常识矛盾
- 声称"技术领先"但无具体性能指标或第三方验证
- 核心技术依赖开源项目，无自研壁垒
- 研发团队规模与声称的技术复杂度不匹配

### 第四步：关键问题提出

基于盲区识别和交叉验证的结果，提出投决必知的关键问题。

**问题配比**：技术 30% / 商业 20% / 其他内容 30% / 盲区 20%

问题框架参考：`references/tech-question-framework.md`

#### 4.1 技术问题（30%）

针对技术维度的深度验证问题：

1. **技术架构验证**：核心技术栈是什么？架构设计如何支撑扩展？关键组件的自研 vs 开源比例？
2. **性能指标核实**：声称的性能指标在什么条件下测试？与竞品/行业基准对比如何？有无第三方验证？
3. **技术壁垒真实性**：壁垒的具体形式是什么（专利/数据/算法/工程经验）？壁垒可持续多久？被绕过的可能性？
4. **技术路线风险**：当前技术路线的替代方案？开源替代成熟度？大厂跟进可能性？技术演进方向？
5. **研发能力评估**：研发团队规模与背景？研发投入占比？核心技术人才密度？CTO/技术负责人履历？

#### 4.2 商业问题（20%）

针对商业模式和财务的验证问题：

1. **商业模式可持续性**：收入模式是否可持续？客户付费意愿的驱动因素？定价权来源？
2. **客户集中度**：前5大客户收入占比？客户留存率？新客获取成本和渠道？
3. **收入质量**：收入确认方式？递延收入趋势？合同负债变化？

#### 4.3 内容问题（30%）

基于交叉验证发现的问题：

1. **信息一致性**：第三步发现的不一致之处，需要管理层解释
2. **数据可靠性**：关键数据的来源和验证方式
3. **逻辑闭环**：论述中的逻辑跳跃或未闭合的推理链

#### 4.4 盲区问题（20%）

基于第二步识别的盲区：

1. **资料未涉及的关键领域**：盲区识别中高严重性但完全未提及的领域
2. **行业必知但缺失的信息**：行业关键指标缺失对应的问题

#### 问题格式

每个问题按以下格式输出：

```
### Q{n}. {问题标题}

**类别**：技术/商业/内容/盲区
**严重性**：高/中/低
**背景**：资料中关于此问题的现有信息（如有）
**问题**：具体的提问内容
**为什么重要**：此问题对投资决策的影响
**建议追问方向**：如果初步回答不充分，后续追问的角度
```

**问题优先级排序**：
1. 高严重性 + 高影响 → 必须回答
2. 高严重性 + 中影响 → 应当回答
3. 中严重性 + 高影响 → 应当回答
4. 其他 → 可选回答

## 输出契约

最终输出包含以下部分：

1. **盲区识别报告**：所有检测到的盲区，按严重性排序
2. **交叉验证报告**：发现的不一致、证据不足、红旗信号
3. **关键问题清单**：按配比和优先级排序的问题列表
4. **问题汇总表**：所有问题的摘要表格（序号、类别、严重性、标题）

**输出方式**：
- 本地模式：写入 Markdown 文件到资料目录同级
- 飞书协作模式：写入飞书云文档（追加到已有文档或创建新文档），同时本地保存备份

---

## 注意事项

- 盲区识别脚本是纯文本分析，不调用 LLM，结果作为辅助参考
- 交叉验证需要结合行业知识和常识判断，不能仅依赖文本匹配
- 技术问题需要根据公司实际技术领域调整，不是所有技术问题都适用
- 问题数量建议控制在 15-25 个，过多会降低每个问题的深度
- 问题的价值在于"投决必知"，而非"面面俱到"
